# Layer1 到 Layer2 数据传输模拟测试方案 (Layer1 to Layer2 Data Transmission Mock Test Plan)

## 为什么做 (Why)

当前系统需要验证感知层（Layer1）向应用层（Layer2）的数据传输功能是否正常工作。由于真实环境部署复杂且难以控制各种边界场景，需要通过模拟测试方案来：

1. **验证数据传输协议**的正确性和完整性
2. **测试各种边界场景**（正常数据、异常数据、极限值）
3. **模拟真实网络环境**（延迟、丢包、重传）
4. **自动化测试执行**，提高测试效率和覆盖率
5. **提供详细的测试报告**，包括性能指标和错误统计

## 核心变更 (What Changes)

- 创建 Layer1 Mock 数据源，模拟感知层输出
- 创建 Layer2 Mock 接收器，验证数据接收
- 实现数据传输协议模拟器
- 开发自动化测试框架
- 实现终端测试结果展示
- 提供测试覆盖率报告和性能基准对比

## 影响范围 (Impact)

- 受影响的规范：遵循 `docs/data_structures.md` 和 `Android_app/apk_sdk_architecture_guide.md`
- 受影响的代码：新增 `Android_app/features/perception/app-test-perception/src/test/java/` 目录下的测试代码
- 依赖关系：依赖 `module-perception` 和 `module-perception-api` SDK

## 新增需求 (ADDED Requirements)

### 需求 1: Mock 数据源创建
系统必须提供一个符合 Layer1 输出格式的 Mock 数据源。
#### 场景: 生成正常数据
- **WHEN** 测试框架请求生成测试数据
- **THEN** 系统应生成符合 `StandardizedSignals` 格式的完整数据
- **AND** 数据应包含所有必需字段（timestamp、signals、confidence）
- **AND** 数据值应在合理范围内

#### 场景: 生成边界值数据
- **WHEN** 测试框架请求生成边界测试数据
- **THEN** 系统应生成包含极限值的数据
- **AND** 包括最大值、最小值、零值、负值（如适用）
- **AND** 验证接收端能正确处理这些边界值

#### 场景: 生成异常数据
- **WHEN** 测试框架请求生成异常测试数据
- **THEN** 系统应生成包含异常情况的数据
- **AND** 包括缺失字段、空值、格式错误、类型错误
- **AND** 验证接收端能正确识别和处理这些异常

### 需求 2: Layer2 Mock 接收器
系统必须提供一个 Mock 接收器来捕获和验证传输的数据。
#### 场景: 接收并验证数据
- **WHEN** Layer1 发送数据到 Layer2
- **THEN** Mock 接收器应捕获完整的数据
- **AND** 验证数据格式是否符合 `StandardizedSignals` 规范
- **AND** 验证所有必需字段是否存在
- **AND** 验证数据值的类型和范围

#### 场景: 记录传输指标
- **WHEN** 数据传输发生时
- **THEN** Mock 接收器应记录传输指标
- **AND** 包括传输时间戳、数据大小、传输延迟
- **AND** 记录成功/失败状态

### 需求 3: 数据传输协议模拟
系统必须模拟真实的数据传输协议，包括网络延迟、丢包、重传等场景。
#### 场景: 模拟正常传输
- **WHEN** 模拟器处于正常传输模式
- **THEN** 数据应在配置的刷新间隔内成功传输
- **AND** 传输延迟应在合理范围内（< 100ms）
- **AND** 不应发生丢包或重传

#### 场景: 模拟网络延迟
- **WHEN** 模拟器配置了网络延迟
- **THEN** 数据传输应延迟指定的时间
- **AND** 接收端应记录实际延迟时间
- **AND** 延迟时间应在配置的误差范围内（±10%）

#### 场景: 模拟丢包
- **WHEN** 模拟器配置了丢包率
- **THEN** 部分数据包应丢失
- **AND** 丢失率应接近配置值（±5%）
- **AND** 接收端应记录丢包事件

#### 场景: 模拟重传机制
- **WHEN** 检测到数据包丢失
- **THEN** 模拟器应自动重传丢失的数据包
- **AND** 重传应在合理时间内完成（< 500ms）
- **AND** 最大重传次数应可配置（默认 3 次）

### 需求 4: 自动化测试框架
系统必须提供一个自动化测试框架，支持批量测试用例执行。
#### 场景: 执行测试套件
- **WHEN** 测试框架执行测试套件
- **THEN** 系统应按顺序执行所有测试用例
- **AND** 每个测试用例应有独立的测试环境
- **AND** 测试结果应独立记录

#### 场景: 并行测试执行
- **WHEN** 测试框架配置了并行执行
- **THEN** 独立的测试用例应并行执行
- **AND** 并行度应可配置（默认 4 个并发）
- **AND** 测试结果应正确关联到对应的测试用例

#### 场景: 测试失败处理
- **WHEN** 测试用例执行失败
- **THEN** 系统应记录详细的失败信息
- **AND** 包括失败原因、堆栈跟踪、相关日志
- **AND** 不应影响其他测试用例的执行

### 需求 5: 终端测试结果展示
系统必须在终端内清晰展示每项测试的通过状态、性能指标、错误详情等关键信息。
#### 场景: 显示测试进度
- **WHEN** 测试框架执行测试套件
- **THEN** 终端应实时显示测试进度
- **AND** 显示当前执行的测试用例名称
- **AND** 显示已完成的测试用例数量和总数
- **AND** 显示预计剩余时间

#### 场景: 显示测试结果
- **WHEN** 测试用例执行完成
- **THEN** 终端应显示测试结果
- **AND** 使用颜色标识通过/失败状态（绿色/红色）
- **AND** 显示测试用例名称、状态、执行时间
- **AND** 显示性能指标（传输延迟、数据大小、成功率）

#### 场景: 显示错误详情
- **WHEN** 测试用例执行失败
- **THEN** 终端应显示详细的错误信息
- **AND** 包括错误类型、错误消息、堆栈跟踪
- **AND** 提供错误定位和建议

#### 场景: 显示测试摘要
- **WHEN** 所有测试用例执行完成
- **THEN** 终端应显示测试摘要
- **AND** 包括总测试用例数、通过数、失败数、跳过数
- **AND** 计算并显示通过率
- **AND** 显示平均执行时间和性能指标

### 需求 6: 测试覆盖率报告
系统必须提供测试覆盖率报告和性能基准对比功能。
#### 场景: 生成覆盖率报告
- **WHEN** 测试框架执行完成
- **THEN** 系统应生成测试覆盖率报告
- **AND** 包括代码覆盖率百分比
- **AND** 包括功能覆盖率（已测试功能 / 总功能数）
- **AND** 标识未覆盖的代码路径和功能点

#### 场景: 性能基准对比
- **WHEN** 测试框架执行性能测试
- **THEN** 系统应记录性能基准数据
- **AND** 包括传输延迟、吞吐量、资源使用
- **AND** 与历史基准数据进行对比
- **AND** 标识性能回归或改进

#### 场景: 生成测试报告
- **WHEN** 测试框架执行完成
- **THEN** 系统应生成详细的测试报告
- **AND** 报告格式应为 Markdown 或 HTML
- **AND** 包括测试摘要、详细结果、覆盖率分析、性能对比
- **AND** 报告应可导出为文件

### 需求 7: 可扩展性设计
系统必须具备可扩展性，支持后续新增测试场景和协议版本升级。
#### 场景: 添加新测试场景
- **WHEN** 需要添加新的测试场景
- **THEN** 系统应支持通过配置文件或代码扩展添加
- **AND** 不应修改核心框架代码
- **AND** 新场景应能无缝集成到现有测试套件

#### 场景: 协议版本升级
- **WHEN** 数据传输协议升级到新版本
- **THEN** 系统应支持多版本协议
- **AND** 应能自动识别协议版本
- **AND** 应能根据版本选择相应的解析器
- **AND** 向后兼容旧版本协议

#### 场景: 自定义 Mock 数据生成器
- **WHEN** 需要自定义 Mock 数据生成逻辑
- **THEN** 系统应支持插件式数据生成器
- **AND** 应提供统一的接口定义
- **AND** 应能动态加载和卸载生成器

## 技术实现要点

### Mock 数据源设计
- 使用工厂模式创建不同类型的 Mock 数据生成器
- 支持随机数据生成和固定数据集
- 实现数据验证器确保生成的数据符合规范
- 支持数据序列化和反序列化（JSON）

### Mock 接收器设计
- 使用观察者模式监听数据传输事件
- 实现数据验证器检查数据完整性
- 记录传输指标（时间戳、延迟、大小）
- 支持数据持久化用于后续分析

### 传输协议模拟器设计
- 使用策略模式支持不同的传输场景
- 实现延迟注入器模拟网络延迟
- 实现丢包模拟器模拟网络不稳定
- 实现重传机制处理丢包情况

### 自动化测试框架设计
- 使用测试套件模式组织测试用例
- 支持测试用例的依赖关系管理
- 实现测试运行器支持串行和并行执行
- 提供测试监听器接口用于自定义报告

### 终端输出设计
- 使用 ANSI 颜色代码增强可读性
- 实现进度条显示测试进度
- 使用表格格式展示测试结果
- 支持详细模式和摘要模式切换

### 测试报告生成
- 使用模板引擎生成 Markdown/HTML 报告
- 实现覆盖率计算器
- 实现性能对比器
- 支持报告导出为多种格式

## 用户体验设计

### 测试执行流程
1. 配置测试参数（测试套件、并发度、传输协议）
2. 启动测试框架
3. 实时查看测试进度和结果
4. 测试完成后查看详细报告
5. 根据报告分析问题并优化

### 终端界面设计
- 使用清晰的分区布局（配置区、进度区、结果区）
- 关键信息使用醒目的颜色（通过=绿色，失败=红色）
- 支持交互式命令（暂停、继续、跳过）
- 提供快捷键支持（Ctrl+C 中断，空格暂停）

### 报告展示设计
- 使用层级结构（摘要 → 详细结果 → 覆盖率 → 性能对比）
- 使用图表展示性能趋势
- 使用表格展示测试结果对比
- 提供搜索和过滤功能

## 测试与验证

### 单元测试
- 测试 Mock 数据生成器的各种场景
- 测试数据验证器的准确性
- 测试传输协议模拟器的正确性
- 测试测试框架的核心逻辑

### 集成测试
- 测试 Layer1 Mock 数据源与真实 SDK 的兼容性
- 测试 Layer2 Mock 接收器与真实接收端的兼容性
- 测试端到端的数据传输流程
- 测试各种网络场景下的数据传输

### 性能测试
- 测试大量数据传输的性能
- 测试高并发下的稳定性
- 测试长时间运行的内存泄漏
- 测试不同传输协议的性能差异

### 验收测试
- 验证所有需求是否满足
- 验证测试覆盖率是否达到目标（> 80%）
- 验证测试报告的完整性和准确性
- 验证终端输出的可读性和实用性
