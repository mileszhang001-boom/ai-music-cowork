# 文档整合指南 (Documentation Integration Guide)

## 1. 整合说明报告
为了提升项目文档的可读性、可维护性以及团队协作效率，我们对原有的文档体系进行了重构和整合。

### 变更摘要
- **保留并优化**：`constitution.md`、`spec-all.md`、`plan.md`。为这些文档添加了目录（TOC）和交叉引用，使其结构更加清晰。
- **重命名并优化**：
  - 原 `rd_tasks.md` 重命名为 `tasks.md`，作为研发任务拆解的专属文档。
  - 原 `testing_acceptance.md` 重命名为 `testing.md`，作为核心场景测试与验收的专属文档。
- **合并**：
  - `tech.md` 的内容已合并至 `plan.md`，作为“全链路技术方案评估与选型”章节。
  - `checklist.md` 的内容已合并至 `testing.md`，作为“核心检查项 Checklist”章节。
- **新增**：
  - `README.md`：作为项目的入口，提供项目简介、文档导航和 GitHub 协作指南。
  - `doc_integration_guide.md`：即本文档，说明文档体系的变更和使用方法。
- **删除**：清理了原有的 `rd_tasks.md` 和 `testing_acceptance.md`，避免内容冗余。

### 核心文档定位
1. **`constitution.md`**：高层级的项目宪法，适合新成员快速了解项目愿景、原则和约束。
2. **`spec-all.md`**：最详尽的“字典”，包含所有的产品逻辑、数据结构和技术规范。遇到细节问题时查阅此文档。
3. **`plan.md`**：架构与计划的结合，展示系统如何运转（架构图）以及整体的推进节奏。
4. **`tasks.md`**：执行层面的任务清单，用于项目管理和进度追踪。
5. **`testing.md`**：质量保证的基准，用于指导测试用例的编写和最终的验收。

## 2. 文档使用指南

### 如何查找信息？
- **想了解项目要做什么？** -> 阅读 `README.md` 和 `constitution.md`。
- **想知道某个 JSON 字段的含义？** -> 查阅 `spec-all.md` 的 [数据结构设计](spec-all.md#6-数据结构设计) 章节。
- **想了解系统各模块如何交互？** -> 查看 `plan.md` 的 [技术架构](plan.md#3-技术架构) 章节中的架构图。
- **想知道本周需要完成什么开发任务？** -> 查看 `tasks.md` 中对应的 Phase。
- **开发完成后如何验证？** -> 参考 `testing.md` 中的场景走查路径。

### 文档维护原则
- **单一事实来源 (Single Source of Truth)**：避免在多个文档中重复定义相同的内容。例如，数据结构只在 `spec-all.md` 中定义，其他文档通过链接引用。
- **同步更新**：当系统架构或接口发生变更时，必须同步更新相关的文档，并在 Commit Message 中注明 `docs: update xxx`。
- **保持交叉引用有效**：在修改文档标题或结构时，注意检查并更新其他文档中指向该部分的链接。
