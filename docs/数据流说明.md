# 车载 AI 音乐推荐系统 - 数据流说明

> 一句话概括：**系统通过传感器感知你的驾驶状态，用模板快速响应，同时让 AI 在后台优化，给你"先快后精"的体验。**

---

## 整体流程图

```
┌─────────────────────────────────────────────────────────────────────┐
│                         用户驾驶场景                                 │
│            (车速、天气、疲劳度、乘客数量、语音指令...)                   │
└─────────────────────────────────────────────────────────────────────┘
                                  ↓
┌─────────────────────────────────────────────────────────────────────┐
│  第一步：感知层 (Layer 1)                                            │
│  ─────────────────────                                              │

│  • 输入：车速 72km/h、天气"晴"、疲劳度 0.3...                        │
│  • 输出：标准化数据 (全部归一化到 0~1 范围)                          │
│  • 耗时：< 5 毫秒 (纯本地计算)                                       │
└─────────────────────────────────────────────────────────────────────┘
                                  ↓
┌─────────────────────────────────────────────────────────────────────┐
│  第二步：语义层 (Layer 2) - 核心决策层                               │
│  ────────────────────────────────                                   │
│  理解当前是什么场景，决定怎么响应                                     │
│                                                                      │
│  【快通道】模板匹配 (毫秒级)                                         │
│  ├─ 从预设模板库中匹配最合适的场景模板                               │
│  ├─ 比如：早晨通勤 → 播放轻快音乐 + 暖色灯光                         │
│  └─ 立即执行！用户马上感受到变化                                     │
│                                                                      │
│  【慢通道】AI 推理 (秒级，后台异步)                                  │
│  ├─ 调用阿里云通义千问大模型                                        │
│  ├─ 根据更多上下文生成个性化推荐                                    │
│  └─ 结果返回后，平滑过渡更新                                        │
└─────────────────────────────────────────────────────────────────────┘
                                  ↓
┌─────────────────────────────────────────────────────────────────────┐
│  第三步：效果层 (Layer 3) - 执行层                                   │
│  ─────────────────────────────                                       │
│  把决策变成实际的音效、灯光、音乐                                    │
│  • 🎵 音乐引擎：生成歌单、播放音乐                                   │
│  • 💡 灯光引擎：调整氛围灯颜色和亮度                                 │
│  • 🔊 音效引擎：调整均衡器、空间音频等                              │
└─────────────────────────────────────────────────────────────────────┘
```

## 核心机制：快慢双通道

### 为什么需要双通道？

| 场景 | 问题 |
|------|------|
| 只用模板 | 响应快，但不够智能，无法处理复杂需求 |
| 只用 AI | 很智能，但要等 3-8 秒，用户体验差 |
| **双通道** | **模板先响应 (10ms)，AI 后台优化 (3-8s)，两全其美** |

### 快通道 (Fast Track)

```
触发条件：任何场景变化
处理方式：本地模板匹配
响应时间：< 10 毫秒

工作流程：
1. 从 50+ 预设模板中匹配
2. 匹配规则：时间 + 天气 + 车速 + 乘客数 + 疲劳度
3. 命中模板 → 立即执行
```

**预设模板示例：**

| 模板名称 | 触发条件 | 音乐风格 | 灯光 | 音效 |
|---------|---------|---------|------|------|
| 早晨通勤 | 工作日早晨、独自驾车 | 轻快流行 | 暖白 | 清晰人声 |
| 深夜驾驶 | 23:00-5:00 | 舒缓爵士 | 暗蓝 | 夜间模式 |
| 疲劳提醒 | 疲劳度 > 70% | 提神摇滚 | 明亮 | 增强高音 |
| 朋友出游 | 乘客 ≥ 2 人 | 欢快流行 | 彩色 | 派对模式 |
| 雨夜行车 | 下雨 + 夜间 | 轻音乐 | 柔和 | 降噪模式 |

### 慢通道 (Slow Track)

```
触发条件：快通道启动后，异步执行
处理方式：调用阿里云通义千问 (Qwen) 大模型
响应时间：3-8 秒 (普通模式) / 8-15 秒 (深度思考模式)

工作流程：
1. 构建提示词 (包含场景上下文 + 快通道模板)
2. 调用云端大模型
3. 解析 AI 返回的场景描述
4. 计算与快通道结果的差异
5. 只更新有变化的部分 (平滑过渡)
```

**AI 能做什么？**

- 理解模糊需求："帮我放点适合现在的歌"
- 处理复杂约束："不要太吵，但也不能太困"
- 个性化推荐：结合用户历史偏好
- 生成解释：为什么推荐这首歌

## 语音指令处理

### 快速旁路 (Bypass)

对于简单控制命令，直接响应，不走完整流程：

| 用户说 | 系统响应 | 延迟 |
|-------|---------|------|
| "暂停" | "好的，已暂停" | ~0ms |
| "继续" | "好的，继续播放" | ~0ms |
| "下一首" | "好的，下一首" | ~0ms |
| "声音大点" | "好的，已调高音量" | ~0ms |

### 意图分类

| 意图类型 | 关键词 | 预估等待 |
|---------|-------|---------|
| 控制 | 暂停、继续、调大、调小 | 1 秒 |
| 导航 | 导航、去、路线 | 3 秒 |
| 查询 | 什么、是谁、多少 | 5 秒 |
| 创作 | 播放、来首、想听、帮我选 | 8 秒 |

## 歌单生成算法

音乐引擎采用 **三级瀑布策略**：

```
┌────────────────────────────────────────┐
│ 第一级：预设音乐库                      │
│ ──────────────────                     │
│ 场景在预设列表中？                      │
│ ├─ 是 → 从音乐库加载预定义曲目          │
│ └─ 否 → 进入下一级                     │
│                                        │
│ 延迟：< 5ms                             │
└────────────────────────────────────────┘
                 ↓
┌────────────────────────────────────────┐
│ 第二级：AI 大模型生成                   │
│ ───────────────────────                │
│ 调用通义千问，模拟全网搜索              │
│ 返回带情感解释的个性化歌单              │
│                                        │
│ 延迟：3-8 秒                            │
└────────────────────────────────────────┘
                 ↓
┌────────────────────────────────────────┐
│ 第三级：本地 Mock 数据 (兜底)           │
│ ──────────────────────────             │
│ 使用 8 首硬编码的模拟曲目               │
│                                        │
│ 延迟：< 1ms                             │
└────────────────────────────────────────┘
```

**AI 生成的歌曲信息包含：**

- 基本信息：歌名、歌手、流派、时长
- 音乐特征：BPM、能量值
- 情感解释：为什么适合当前场景

## 延迟汇总

| 场景 | 通道 | 延迟 |
|------|------|------|
| 简单控制命令 (暂停/下一首) | 快速旁路 | **~0ms** |
| 模板命中 (常见场景) | 快通道 | **< 10ms** |
| AI 缓存命中 | 慢通道缓存 | **< 5ms** |
| AI 普通推理 | 慢通道 | **3-8s** |
| AI 深度思考 | 慢通道 | **8-15s** |
| AI 失败降级 | Fallback | **超时后 0ms** |

## 完整端到端流程示例

**场景：周五晚上，用户开车回家，说了一句"帮我放点适合现在的歌"**

```
时间线：
─────────────────────────────────────────────────────────────────→

T+0ms    用户说话："帮我放点适合现在的歌"
         ↓
T+5ms    感知层：识别语音输入，提取时间(21:00)、车速(60km/h)等信号
         ↓
T+10ms   语义层-快通道：匹配到"深夜驾驶"模板
         ├─ 音乐：舒缓爵士
         ├─ 灯光：暗蓝色呼吸
         └─ 音效：夜间模式
         ↓
T+15ms   效果层：立即执行模板配置
         ├─ 播放预设歌单
         └─ TTS："已为您切换到深夜驾驶模式"
         
         用户已经听到音乐了！✅
         
         (同时，后台启动慢通道...)
         ↓
T+3000ms 语义层-慢通道：AI 返回个性化推荐
         ├─ 分析：用户周五晚上下班，可能有些疲惫但心情放松
         ├─ 推荐：轻爵士 + 几首用户常听的轻音乐
         └─ 差异：比模板多了一些用户偏好的曲目
         ↓
T+3500ms 效果层：平滑过渡更新歌单
         └─ 当前歌曲播完后，切换到 AI 推荐的歌单
         
         用户感受到的是无缝的个性化体验！✅
```

## 总结

这个系统的设计理念是 **"先快后精"**：

1. **快通道**：用模板保证基础体验，10ms 内响应
2. **慢通道**：用 AI 提供个性化，后台异步优化
3. **平滑过渡**：AI 结果到达后，只更新有变化的部分

用户感受到的是：**说一句话，音乐立刻响起，几秒后变得更懂我**。
