## Context

### 背景

当前 Layer 3 采用单一的编排协调器模式，直接调用 Content/Lighting/Audio 引擎执行任务。随着场景复杂度增加，存在以下问题：

1. **任务分析不够深入**：单一编排器缺乏对任务意图的深度理解能力
2. **执行监控不足**：缺乏对子任务执行过程的实时监控和反馈机制
3. **质量检验缺失**：没有标准化的执行结果检验流程
4. **扩展性受限**：难以支持复杂的任务拆分和多轮交互

### 当前架构

```
Scene Descriptor → Orchestrator → Content/Lighting/Audio Engines → EffectCommands
```

### 约束条件

- Scene Descriptor 输入格式保持不变（向后兼容）
- Effect Commands 输出格式保持不变
- 需要复用现有 LLM 客户端和事件总线
- 响应时间要求：<500ms（主Agent分析时间）

## Goals / Non-Goals

**Goals:**

1. 实现主Agent + 3个Sub-Agent的多Agent架构
2. 主Agent具备任务分析、决策和分发的能力，核心指标：分析准确≥95%、命中率≥90%、一次成功率≥85%
3. Sub-Agent采用小步快跑模式，每分钟汇报一次执行情况
4. 所有Agent采用ReAct架构，具备反思和自检能力
5. 建立标准化的Agent通信协议

**Non-Goals:**

- 不实现Agent之间的P2P通信（统一通过主Agent协调）
- 不实现Agent的动态注册/发现（预定义3个子Agent）
- 不实现跨Layer的Agent通信（仅Layer 3内部）
- 不实现Agent的持久化状态存储（内存存储）

## Decisions

### D1: 主Agent架构选择

**决策**：采用"分析-决策-分发-检验"四阶段流程

**理由**：
- 符合ReAct架构的Thought/Action模式
- 清晰的职责分离，便于监控和调试
- 与现有编排器兼容性好

**备选方案**：
- 单一Agent直接分发 → 缺乏深度分析能力
- 并行同时调用所有子Agent → 无法处理依赖关系

### D2: Sub-Agent汇报机制

**决策**：定时汇报 + 事件触发双通道

**汇报频率**：每分钟汇报一次
**事件触发**：任务完成/失败/需要决策时立即汇报

**理由**：
- 定时汇报确保主Agent了解整体进度
- 事件触发确保关键信息及时传达
- 平衡了通信开销和实时性需求

**备选方案**：
- 仅事件触发 → 可能遗漏中间状态
- 仅定时汇报 → 实时性不足

### D3: ReAct实现方式

**决策**：基于LLM的ReAct循环 + 规则引擎兜底

**理由**：
- LLM提供强大的推理能力
- 规则引擎作为兜底，确保关键决策的可预测性
- 符合车载系统的安全要求

**备选方案**：
- 纯LLM ReAct → 不可预测，可能产生安全风险
- 纯规则引擎 → 缺乏灵活性

### D4: 任务状态管理

**决策**：主Agent维护全局任务状态图

```
状态图结构：
{
  task_id: {
    status: pending|running|completed|failed,
    assignee: sub-agent-id,
    progress: 0-100,
    result: any,
    history: [{action, observation, timestamp}]
  }
}
```

**理由**：
- 集中管理便于主Agent进行全局优化
- 历史记录支持回溯和调试
- 便于计算核心指标

### D5: 指标收集策略

**决策**：异步批量收集 + 关键节点标记

**理由**：
- 减少对主执行流程的影响
- 支持事后分析和优化
- 关键节点标记确保重要指标不丢失

## Risks / Trade-offs

### R1: LLM调用延迟

**风险**：主Agent的ReAct循环可能需要多次LLM调用，导致响应延迟

**缓解**：
- 设置最大循环次数（默认3次）
- 规则引擎兜底，减少LLM调用
- 缓存常用场景的决策结果

### R2: Sub-Agent同步问题

**风险**：多个Sub-Agent并发执行时可能产生状态不一致

**缓解**：
- 主Agent作为唯一状态写入源
- 使用乐观锁处理并发更新
- 定期状态校验机制

### R3: 任务分发失败

**风险**：Sub-Agent执行失败导致整体任务失败

**缓解**：
- 主Agent具备任务重试能力
- 降级策略：部分成功也返回结果
- 详细的错误分类和恢复策略

### R4: 通信协议复杂性

**风险**：Agent间通信协议过于复杂增加开发难度

**缓解**：
- 简化协议格式，采用JSON
- 预定义消息类型，减少动态解析
- 提供调试工具支持协议追踪

1. 创建 `src/core/agents/` 目录结构
2. 实现ReAct框架核心
3. 实现Agent通信协议
4. 单元测试覆盖

### Phase 2: 主Agent（Week 2）

1. 实现任务分析模块
2. 实现任务分发逻辑
3. 实现执行结果检验
4. 集成现有LLM客户端

### Phase 3: Sub-Agent（Week 3）

1. 实现Content Agent
2. 实现Lighting Agent
3. 实现Audio Agent
4. 实现汇报机制

### Phase 4: 集成测试（Week 4）

1. 主Agent与Sub-Agent联调
2. 与现有Layer 3集成
3. 端到端测试
4. 性能优化

### 回滚策略

- Phase 1/2失败：回退到原有编排器模式
- Phase 3失败：Sub-Agent降级为直接函数调用
- Phase 4失败：保留原有入口，新逻辑作为可选路径

   - 当前设计：内存存储
